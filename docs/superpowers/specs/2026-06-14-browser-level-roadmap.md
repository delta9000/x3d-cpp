## Hang root-cause (B1 — empirically verified)

- **Root cause:** BoundsSystem::compute() (runtime/scene/BoundsSystem.hpp) is a recursive post-order DFS over the node graph with NO memoization. When a node is reachable by many distinct root->node paths — which happens whenever a node instance is shared under multiple parents (shared Appearance/Material/ImageTexture/geometry sub-nodes, USE'd nodes, or parser-reused node instances) — compute() recomputes that node AND its entire subtree once PER PATH. With ~154 nested Transforms/Groups and shared appearance nodes, the number of root->node paths is exponential in nesting depth, so compute() makes astronomically many calls and never returns (100% CPU, flat RSS — no allocation growth because local_ is bounded; it is pure re-walk). The companion index() DFS (line 57) shares the same no-visited-set design and pushes duplicate child edges into children_, which both feeds the blowup and is a secondary (currently non-fatal) hardening gap. NOTE: this is NOT a cyclic ROUTE graph, NOT Inline recursion, NOT proto expansion, and NOT a true cycle — DirtBike.x3d has zero DEF/USE/Inline/Proto. It is exponential shared-subgraph re-walk.
- **Location:** runtime/scene/BoundsSystem.hpp :: BoundsSystem::compute() (recursive call at line 94, invoked from buildBounds() line 28-29, which runs inside X3DExecutionContext::buildSceneGraph() at runtime/events/X3DExecutionContext.hpp:61). Same unbounded-DFS pattern also in BoundsSystem::index() (line 57).  **Mechanism:** Exponential path re-walk, not an infinite cycle. index() builds children_ adjacency where a shared node appears as a child under every parent that references it (multiply-parented + duplicate edges). compute(root) then recurses into each child via children_[n]; a shared node S with subtree below it is fully recomputed once for every distinct root->S path. Confirmed by instrumentation: on DirtBike a single shared Material node was entered by compute() 1000 times within only the first 1129 total compute calls; on Church a shared ImageTexture node hit 1000 entries within 3497 calls. The per-node visit count grows multiplicatively with each additional Transform nesting level that has >1 path to the shared node, so total compute() invocations explode well beyond any finite bound.
- **Fix:** Memoize compute(): at the top of BoundsSystem::compute(), if n already has a local_ entry from this buildBounds() pass, return it immediately instead of re-walking. Because buildBounds() clears local_ at the start and local bounds are computed in each node's OWN frame (the parent applies the child Transform's localMatrix at the union site, line 95), the first-path value is correct for every subsequent path, so a plain `auto it = local_.find(n); if (it != local_.end()) return it->second;` guard at the function entry is sound and collapses the work to O(distinct nodes). I validated exactly this guard in a temporary build: DirtBike went from infinite hang to 719 render items in ~14 ms, Church from infinite hang to 2573 render items in ~48 ms, both rc=0. RECOMMENDED HARDENING (secondary, same patch): also guard index() with a visited set so children_ is built over distinct edges only (avoid duplicate child entries and bound the index pass), and consider adding a defensive recursion depth cap as belt-and-suspenders against any genuinely cyclic graph a future input could present. The minimal one-line memo guard alone fixes the reported hang; the index() visited-set is the clean complete form.  (complexity: small, confidence: high)
- **Regression test:** 1) Add a timing-bounded extraction test (new gtest in runtime/extract/tests or runtime/scene/tests) that loads DirtBike.x3d and SanCarlosCathedral/Church.x3d, calls ctx.buildSceneGraph(scene) + a fullSnapshot extraction, and asserts it completes under a wall-clock bound (e.g. 5 s) and yields the expected non-zero render-item counts (DirtBike 719, Church 2573 with the current extractor). The test FAILS (times out) on the pre-fix code and passes after. 2) Add a focused BoundsSystem unit test constructing a small shared-subgraph (one Group whose children list contains the SAME shared child shared_ptr N times, nested a few Transform levels deep) and assert buildBounds() returns quickly and that compute() is called once per distinct node (e.g. via a visit-count probe or simply a tight time bound) — this directly pins the exponential-rewalk regression independent of the corpus files. 3) Keep the existing BoundsSystem golden/bounds tests green to confirm memoization does not change computed AABB values (local bounds are frame-local, so values must be identical). 4) Optionally extend the 16-scene usability gate to assert all scenes extract within a per-scene time budget.

## Definition of browser-level

For THIS renderer, browser-level is parity-of-coverage and never-freeze measured against X3DOM, X_ITE and Castle on the conformance corpus at $X3D_CORPUS_DIR, NOT photorealism, gated by: (1) ROBUSTNESS never-freeze: every corpus scene completes parseFile plus buildSceneGraph plus a full extraction within a bounded wall-clock budget (target under 5s per scene, typically under 100ms); zero hangs. EMPIRICALLY-PINNED acceptance: the four named hang files (Chess8Levels, DiamondManLOA1, HAnimModelFootLeft, KoreanCharacter09Ru) plus DirtBike and Church each extract non-zero items under the time bound; the test asserts completion-under-bound and itemCount over zero, NOT exact counts (verified values today 714/67/172/80/719/2573 are advisory, may be pinned as a separate post-fix regression check). (2) OBSERVABILITY: no geometry silently dropped; every unsupported or recognized-but-empty geometry node is counted and surfaced at the extraction seam. (3) GEOMETRY COVERAGE: render items for every common authored geometry type (IFS, triangle sets, strips, fans, quads, ElevationGrid, primitives done) PLUS Extrusion, IndexedLineSet, LineSet, PointSet, GeoElevationGrid, Text; lines and points flow through a topology-typed seam and draw via an explicitly-unlit path, not garbage triangles. Target: corpus clean percentage rises from ~59.8% toward over 90%, the GeoElevationGrid unlock flipping the Savage-dominated majority. (4) SHADING QUALITY: creaseAngle smoothing on IFS and grids, Blinn-Phong specular, transparency alpha-blend with back-to-front sort. (5) SCENE FEATURES: textures appear via embedder byte-supply, Background sky/ground gradient, Fog, interactive orbit/examine camera from the bound Viewpoint and NavigationInfo. INVARIANTS: the SDK never does I/O and never owns a window or input device; textures, Inline bytes, fonts, geodesy and input enter through embedder std function seams; the renderer consumes POD descriptors; the generated layer (generated_cpp_bindings) and golden files are byte-identical after every milestone, enforced by a CI gate; RenderItem.hpp is hand-written contract NOT generated, so editing it in B4 is not a golden violation.

- B4 (MeshData Topology) touches the renderer-facing CONTRACT (RenderItem.hpp, HAND-WRITTEN not generated, not a golden violation) and the PoC draw loop simultaneously. Default Triangles keeps existing paths safe, BUT the PoC MUST be updated in the same change or lines and points upload as triangles and render garbage. A naive impl produces twisted or inverted walls; needs golden tests against a known shape (Emma2.x3d hull). AND B3 must emit in lattice-index-retaining form or B6 cannot smooth it.
- B6 creaseAngle welding MUST re-derive adjacency from SOURCE index space (coordIndex or lattice), NOT the expanded 0..N-1 run or float position-keying.
