# Project Wiki — canonical knowledge home (MkDocs), 3-wave build toward 100% coverage

**Date:** 2026-06-20
**Status:** Approved design; build in 3 workflow-orchestrated waves.
**Goal:** A single, human-renderable, agent-navigable knowledge home for x3d-cpp-gen that closes the scattered-knowledge seams (63 unindexed specs/plans, two separate RAGs, conformance view, auto-memory, commits) by giving the *current* "how it is + why" one canonical place — authored markdown, rendered as a browsable site, version-controlled with the code.

## Why
Knowledge is spread across 5+ disconnected systems with no map: `docs/superpowers/specs` (46) + `plans` (17) have no index; `spec_rag`/`code_rag` are separate CLI tools; `docs/conformance` is the only generated zoom-out; the rest lives in auto-memory, the BACKLOG, the SDK README, and commit messages. As the project grows (5 CLI commands + sim this session alone), the seams cost the agent the most — structure is rediscovered every session. One canonical, rendered wiki fixes this.

## Framework & home (decided)
- **MkDocs + Material**, installed via `uv` (a `docs` dependency group), run via `mise run docs` (serve) / `mise run docs-build` (static). Fits the Python/uv/mise stack; no Node/Rust toolchain.
- Source lives in-repo at **`docs/wiki/`** — plain markdown, edited via Read/Edit/Grep, **version-controlled with the code** so a decision moves with what it describes.
- **Cross-linking:** standard relative markdown links (portable, render everywhere). Frontmatter on each page (`title`, `summary`, `tags`, `updated`, `related:` list). Material gives sidebar nav (from `mkdocs.yml`), **offline client-side search** (no self-hosted vector-store / embedding-host dependency), code highlighting, admonitions, dark mode.
- **`mkdocs.yml`** at repo root (or `docs/`), nav tree authored explicitly (discoverable).

## Structure
```
docs/wiki/
  index.md            # the map / "start here" — links to every section + the coverage manifest
  architecture.md     # the system map: generator → bindings → runtime → systems → extract → SDK → CLI;
                      #   the seams (ScriptEngine/AssetResolver/FontMetrics/ext-firewall); data flow
  coverage.md         # the coverage manifest (target set + status) — how "100%" is measured (see below)
  subsystems/         # one page per major functional area (the canonical list lives in coverage.md)
  decisions/          # ADRs — one page per binding decision (the "why"), from a fixed template
  guides/             # operational how-tos (build/mise, the gate system, the RAGs, add-a-command, …)
  knowledge-map.md    # spec_rag / code_rag: what each indexes, when to use which, how to run them
```
**Conventions (authored in Wave 1, then enforced):** an **ADR template** (context / decision / consequences / status / links), a **subsystem-page template** (purpose / key files / interfaces & seams / how it's tested / related specs+ADRs), the frontmatter schema, and the relative-link style. Pages link INTO the dated specs/plans (the historical record) rather than duplicating them.

## Relationship to existing systems (seam resolution)
- **Specs/plans (63) stay** as the dated, immutable historical design record; the wiki is the *current* canonical layer that **links into** them.
- **Conformance view stays** (generated by `conformance_view.py`); the wiki links it, never duplicates it.
- **Auto-memory** stays the agent's private session-persistence; its *durable/architectural* facts **graduate into the wiki** (the shared, rendered, canonical version) and memory entries point at wiki pages. Wave 3 does this graduation pass.
- **RAGs** documented in `knowledge-map.md` and made reliably runnable (`uv run`); a **`project_rag`** over the wiki (third pillar, sibling to spec/code) is a **noted follow-on, not in scope here**.

## "100% coverage" — definition & measurement
Coverage is measured against an explicit **coverage manifest** (`coverage.md`, authored in Wave 1) enumerating the target set, each marked covered/missing:
1. **Subsystem coverage** — every top-level functional area gets a page. Canonical list derived from the architecture layers + directory structure: generator/templates; generated bindings; runtime core (scene graph, reflection, execution context, dirty/bounds/transform); codecs & parse (XML/VRML/JSON readers+writers, canonical, inline-expand); event/behavior runtime (cascade, routes, and each System: Time, Interpolators incl. spline, EventUtility, ViewDependent, ViewpointBind, Navigation, Pointing, KeyDevice, Script/SAI, sensors); extract (SceneExtractor/MeshBuilder/PackedMesh/textures/topology); the ext firewall (binary/STL/ExternalGeometry); the SDK façade; the CLI suite (convert/validate/extract/canonicalize/sim); the gate & conformance infrastructure.
2. **Decision coverage** — an ADR for every binding decision, harvested from auto-memory (the decisions file), the specs, the BACKLOG, and commit history (e.g. event-API-for-inputOnly, throw-on-range, virtual mixins, golden-files-in-git, the ext firewall, compiled-static-lib build, version-inference ladder, canonicalize source-provenance call, sim snapshot-diff-not-observer, the differential-gate-as-regression-gate, …).
3. **Guide coverage** — the operational how-tos: build & mise tasks; the gate system (golden / conformance / cli-gate regression); the RAGs; the conformance campaign; how to add a CLI command; the workflow/subagent discipline; golden regeneration.

**100% = every manifest entry has a page that meets its template.** A Wave-3 **completeness critic** audits the wiki against the manifest (and re-derives the manifest from code/memory to catch anything new), reports gaps, and they get filled or explicitly logged. "Attempting 100%" = drive the matrix to complete; any residual is named in `coverage.md`, not silently dropped.

## The 3 waves (phased to catch issues early; each a workflow)
**Wave 1 — Scaffold + conventions + manifest + vertical slice (DE-RISK).** Stand up MkDocs Material + `mise run docs`/`docs-build` + `mkdocs.yml` nav + `index.md`. Author the **conventions** (ADR + subsystem templates, frontmatter schema, link style) and the **`docs-check`** gate (mkdocs `--strict` build = dead-link/orphan detection, wired toward CI). Produce the **coverage manifest** (`coverage.md` — the full target set, derived from code+memory+specs). Write a **vertical slice**: `architecture.md` + `knowledge-map.md` + 1 ADR + 1 subsystem page + 1 guide — real pages that prove the framework renders, links resolve, the templates work, and the gate catches rot. **REVIEW GATE:** I render + link-check + read the slice and validate conventions before Wave 2.

**Wave 2 — Subsystem + decision coverage (the bulk).** Fill every subsystem page and every ADR from the manifest. Workflow: pipeline over manifest entries — one agent per page, each grounded in `code_rag`/`spec_rag`/the relevant specs/memory, producing a page to template; a verify stage checks each page (accuracy vs. the code, template-completeness, links resolve). **REVIEW GATE:** strict build + spot-read before Wave 3.

**Wave 3 — Guides + completeness pass (close to 100%).** Fill all guides; run the **completeness critic** (audit coverage vs. manifest, re-derive to catch gaps, cross-link/orphan audit); **graduate** durable auto-memory facts into the wiki + back-link memory→wiki; final strict-build + coverage report. Update `coverage.md` to show 100% (or name the logged residual).

## Testing / gates
- **`mkdocs build --strict`** (`mise run docs-build`) must pass — fails on broken internal links / nav orphans. This is the wiki's "golden gate."
- A **coverage check**: every manifest entry maps to an existing page (a small script or the completeness-critic audit).
- Accuracy: Wave-2 verify stage cross-checks each page's claims against the code (via `code_rag`/file reads) — no invented APIs/paths.
- The wiki touches only `docs/wiki/`, `mkdocs.yml`, `mise.toml`, and the `uv` docs dependency group — **zero** change to `generated_cpp_bindings/`, `runtime/`, or `tools/`; `mise run golden` stays zero-drift.

## Success criteria
1. `mise run docs` serves a browsable Material site; `mise run docs-build --strict` passes (no broken links/orphans).
2. `architecture.md`, `knowledge-map.md`, `coverage.md`, the conventions/templates, and the Wave-1 slice exist and render (Wave 1).
3. Every subsystem + every binding decision in the manifest has a page meeting its template (Wave 2).
4. Every guide exists; the completeness critic reports coverage = 100% (or names the logged residual); durable memory facts are graduated (Wave 3).
5. Core untouched (golden zero-drift); the wiki is the single canonical, rendered, navigable knowledge home.
